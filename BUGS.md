## Bug 1 — `babel` property error in `vite.config.ts`

**Severity:** Build-blocking  
**File:** `vite.config.ts`  
**Discovered:** VS Code TypeScript error panel

### Error

```
Object literal may only specify known properties,
and 'babel' does not exist in type 'Options'. ts(2353)
```

### What happened

The initial setup included a `babel` config block inside `react()` based on documentation
for an older version of `@vitejs/plugin-react`:

```ts
// Broken config
react({
  babel: {
    plugins: ["babel-plugin-styled-components"],
  },
});
```

The installed version of `@vitejs/plugin-react` does not expose a `babel` option
in its `Options` type, causing a TypeScript error.

### Fix

Removed the `babel` block entirely. The default `react()` call with no arguments works
correctly for styled-components v6 — the babel plugin is not required.

```ts
// Fixed
export default defineConfig({
  plugins: [react()],
});
```

## Bug 2 — `Property 'colors' / 'fonts' does not exist on type 'DefaultTheme'`

**Severity:** Build-blocking (8 errors)  
**File:** `src/styles/GlobalStyles.ts`  
**Discovered:** VS Code TypeScript error panel

### Error

```
Property 'colors' does not exist on type 'DefaultTheme'. ts(2339)
Property 'fonts' does not exist on type 'DefaultTheme'. ts(2339)
```

(Same error repeated 8 times for different theme property accesses)

### What happened

styled-components ships with an intentionally **empty** `DefaultTheme` interface.
This is by design — the library doesn't know the shape of your theme. When you
write `${({ theme }) => theme.colors.accent}` inside a template literal, TypeScript
reads the empty `DefaultTheme` and has no idea what `colors` or `fonts` are.

### Fix

Created `src/styled.d.ts` — a TypeScript declaration file that extends `DefaultTheme`
with our custom `Theme` type:

```ts
// src/styled.d.ts
import "styled-components";
import { Theme } from "./styles/GlobalStyles";

declare module "styled-components" {
  export interface DefaultTheme extends Theme {}
}
```

With this in place, TypeScript knows the full shape of the theme everywhere in the project
and all 8 errors resolved immediately.

## Bug 3 — `import type` violations (TypeScript strict mode)

**Severity:** Build-blocking  
**Files:** `usePets.ts`, `PetsContext.tsx`, `PetCard.tsx`, `SortControls.tsx`, `utils/index.ts`  
**Discovered:** `tsc --noEmit` output

### Error

```
TS1484: 'Pet' is a type and must be imported using a type-only import
when 'verbatimModuleSyntax' is enabled.
```

### What happened

The `tsconfig.json` generated by Vite sets `"verbatimModuleSyntax": true`, which
requires all type-only imports to use the `import type` syntax. Regular `import { Pet }`
was used throughout — valid JavaScript but rejected by the strict TypeScript config.

### Fix

Changed all type-only imports to use `import type`:

```ts
// Before
import { Pet } from "../types";
import { SortOption } from "../types";

// After
import type { Pet } from "../types";
import type { SortOption } from "../types";
```

Applied in: `usePets.ts`, `PetsContext.tsx`, `PetCard.tsx`, `SortControls.tsx`, `utils/index.ts`.

## Bug 4 — Stale `index.css` import in `main.tsx`

**Severity:** Runtime error (white screen)  
**File:** `src/main.tsx`  
**Discovered:** Browser console on first run

### Error

```
[vite] Failed to resolve import "./index.css" from "src/main.tsx"
```

### What happened

Vite's default React + TypeScript template generates `main.tsx` with `import './index.css'`.
During project setup, `index.css` was deleted because all styles are handled by
styled-components' `GlobalStyles`. The import remained in `main.tsx` pointing to a
file that no longer existed.

### Fix

Removed the stale import from `main.tsx`:

```ts
// Before
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import "./index.css"; // ← deleted file
import App from "./App.tsx";

// After
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import App from "./App";
```

## Bug 5 — "Pet not found" on every detail page navigation

**Severity:** Feature-breaking (entire detail view broken)  
**Files:** `src/utils/index.ts`, `src/components/PetCard.tsx`, `src/pages/PetDetailPage.tsx`  
**Discovered:** Manual testing — clicking "View →" on any pet card

### Symptom

Every click on the "View →" link on any `PetCard` navigated to `/pets/:id` but
always displayed the "Pet not found" error state, even though the pet data was loaded.

### Root Cause

A URL encoding mismatch between how the ID was created and how React Router
returned it from `useParams()`.

**Step-by-step breakdown:**

```
1. PetCard calls getPetId(pet):
   encodeURIComponent("https://storage.googleapis.com/.../dog.jpg")
   → "https%3A%2F%2Fstorage.googleapis.com%2F...%2Fdog.jpg"

2. Link renders as:
   <Link to="/pets/https%3A%2F%2Fstorage.googleapis.com%2F...">

3. React Router AUTOMATICALLY DECODES route params in useParams():
   id = "https://storage.googleapis.com/.../dog.jpg"   ← decoded!

4. getPetFromId() compared:
   encodeURIComponent(p.url)  ===  id
   "https%3A%2F%2F..."        ===  "https://..."        ← NEVER MATCHES

5. Result: pets.find() returns undefined → "Pet not found" shown every time
```

The critical misunderstanding was that `useParams()` transparently decodes
URL-encoded characters, so the encoded value put into the route and the value
read back are in different forms.

### Fix

Replaced `encodeURIComponent` / `decodeURIComponent` with **base64 encoding**.
Base64 output contains only alphanumeric characters plus `+`, `/`, and `=` —
after URL-safe substitutions (`+→-`, `/→_`, `=→removed`), the result contains
nothing that React Router would try to decode.

```ts
// Before (broken)
export function getPetId(pet: Pet): string {
  return encodeURIComponent(pet.url);
}

export function getPetFromId(pets: Pet[], id: string): Pet | undefined {
  return pets.find((p) => encodeURIComponent(p.url) === id);
}

// After (fixed)
export function getPetId(pet: Pet): string {
  return btoa(pet.url)
    .replace(/\+/g, "-")
    .replace(/\//g, "_")
    .replace(/=/g, "");
}

export function getPetFromId(pets: Pet[], id: string): Pet | undefined {
  try {
    const url = atob(id.replace(/-/g, "+").replace(/_/g, "/"));
    return pets.find((p) => p.url === url);
  } catch {
    // atob throws if input is not valid base64 — fail safely
    return undefined;
  }
}
```

The `try/catch` around `atob` is a defensive measure — if someone manually types
a malformed ID in the URL bar, `atob` throws a `DOMException` which is now caught
and returns `undefined` gracefully, showing the "not found" state cleanly.
